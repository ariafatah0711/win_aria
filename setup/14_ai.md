fixed agent
```bash
# Script untuk mencari dan menghapus default model ECC dari semua file command agar agent ECC menggunakan model yang sama dengan agent lainnya.
$dirs = @(
    (Join-Path (Get-Location) ".opencode\commands"),
    (Join-Path $HOME ".opencode\commands"),
    (Join-Path $HOME ".config\opencode\commands")
)

if ($env:OPENCODE_CONFIG_DIR) {
    $dirs += Join-Path $env:OPENCODE_CONFIG_DIR "commands"
}

$dirs = $dirs |
    Where-Object { Test-Path $_ } |
    Sort-Object -Unique

$hits = Get-ChildItem `
    -Path $dirs `
    -Recurse `
    -File `
    -Filter "*.md" `
    -ErrorAction SilentlyContinue |
    Select-String -Pattern '^\s*agent:\s*everything-claude-code:'

$hits | Format-Table Path, LineNumber, Line -AutoSize

$utf8 = New-Object System.Text.UTF8Encoding($false)

$hits.Path |
    Sort-Object -Unique |
    ForEach-Object {
        $file = $_
        $content = [System.IO.File]::ReadAllText($file)

        $fixed = [regex]::Replace(
            $content,
            '(?m)^(\s*agent:\s*)everything-claude-code:',
            '$1'
        )

        if ($fixed -ne $content) {
            Copy-Item $file "$file.bak" -Force
            [System.IO.File]::WriteAllText($file, $fixed, $utf8)
            Write-Host "Fixed: $file"
        }
    }

# Script untuk menghapus default model ECC dari semua file konfigurasi OpenCode agar agent ECC menggunakan model yang sama dengan agent lainnya.
$files = @(
    $env:OPENCODE_CONFIG,
    "$HOME\.opencode\opencode.json",
    "$HOME\.config\opencode\opencode.json",
    (Join-Path (Get-Location) "opencode.json"),
    (Join-Path (Get-Location) ".opencode\opencode.json")
) |
Where-Object { $_ -and (Test-Path $_) } |
Sort-Object -Unique

$utf8 = New-Object System.Text.UTF8Encoding($false)

foreach ($file in $files) {
    Write-Host "Patching: $file"

    Copy-Item $file "$file.bak" -Force

    try {
        $config = Get-Content $file -Raw | ConvertFrom-Json

        # Hapus default model ECC
        $config.PSObject.Properties.Remove("model")
        $config.PSObject.Properties.Remove("small_model")

        # Hapus model override setiap agent
        if ($config.agent) {
            foreach ($agent in $config.agent.PSObject.Properties) {
                $agent.Value.PSObject.Properties.Remove("model")
            }
        }

        $json = $config | ConvertTo-Json -Depth 100
        [System.IO.File]::WriteAllText($file, $json, $utf8)

        Write-Host "Fixed: $file"
    }
    catch {
        Write-Warning "Gagal parse $file : $($_.Exception.Message)"
    }
}
```

# ECC + OpenCode: Commands, Skills, Agents, dan Workflow

## Cek instalasi ECC

Jalankan OpenCode dari dalam folder project:

```powershell
cd E:\path\ke\project
opencode
```

Di dalam OpenCode, ketik `/` lalu cari command berikut:

```text
/plan
/tdd
/security
/code-review
/verify
/orchestrate
```

Tidak perlu menjalankan semuanya. Tujuannya hanya memastikan command tersebut muncul di daftar autocomplete.

### Cek skill yang tersedia

`/skills` belum tentu tersedia sebagai slash command. Gunakan prompt ini:

```text
Tampilkan semua skill yang tersedia melalui skill tool.

Jangan load seluruh isi skill. Tampilkan:
- nama skill
- deskripsi singkat
- kapan skill tersebut sebaiknya digunakan
```

Cek file secara langsung dari PowerShell:

```powershell
$cfg = $env:OPENCODE_CONFIG_DIR

if (-not $cfg) {
    $cfg = "$HOME\.config\opencode"
}

Write-Host "Config directory: $cfg"

Get-ChildItem "$cfg\commands" `
    -Filter "*.md" `
    -ErrorAction SilentlyContinue |
    Select-Object Name

Get-ChildItem "$cfg\skills" `
    -Directory `
    -ErrorAction SilentlyContinue |
    Select-Object Name

Get-ChildItem "$HOME\.agents\skills" `
    -Directory `
    -ErrorAction SilentlyContinue |
    Select-Object Name
```

## Perbedaan command, skill, agent, dan tool

| Komponen | Fungsi                                                        | Contoh                                       |
| -------- | ------------------------------------------------------------- | -------------------------------------------- |
| Command  | Shortcut untuk menjalankan prompt atau workflow tertentu      | `/plan`, `/security`, `/verify`              |
| Skill    | Kumpulan instruksi dan pola kerja yang dimuat saat dibutuhkan | `tdd-workflow`, `security-review`            |
| Agent    | AI dengan peran, context, model, dan permission tertentu      | `explore`, `security-reviewer`               |
| Tool     | Kemampuan yang dapat dieksekusi agent                         | membaca file, menjalankan test, `skill` tool |

Command dapat memanggil agent atau mengaktifkan skill, tetapi tidak semua command otomatis memakai banyak agent.

---

# Penjelasan command utama

## `/plan`

Menganalisis requirement, mencari pola yang sudah ada di codebase, menilai risiko, dan membuat rencana implementasi.

Command ini sengaja berhenti dan menunggu konfirmasi sebelum mengubah code.

Contoh:

```text
/plan Tambahkan fitur reset password menggunakan email OTP
```

Gunakan ketika:

* fitur mengubah banyak file;
* requirement belum jelas;
* ada perubahan database atau arsitektur;
* risiko implementasinya cukup besar.

Tidak cocok ketika lo ingin agent langsung menyelesaikan implementasi tanpa berhenti.

---

## `/security`

Menjalankan security review terhadap code atau fitur tertentu.

Contoh:

```text
/security Periksa implementasi reset password dan endpoint authentication
```

Biasanya memeriksa:

* injection;
* authentication dan authorization;
* password dan token handling;
* secrets;
* input validation;
* access control;
* rate limiting;
* XSS dan CSRF;
* dependency vulnerability;
* kebocoran informasi melalui error dan log.

---

## `/tdd`

Menjalankan workflow test-driven development:

```text
test gagal → implementasi minimal → test lolos → refactor
```

Contoh:

```text
/tdd Implementasikan reset password menggunakan token sekali pakai
```

Cocok untuk:

* fitur baru;
* bug fix;
* endpoint API;
* business logic penting;
* refactoring yang berisiko.

---

## `/verify`

Menjalankan verification loop setelah implementasi:

```text
build → typecheck → lint → test
```

Contoh:

```text
/verify
```

Gunakan sebelum menganggap fitur selesai atau sebelum membuat pull request.

---

## `/code-review`

Memeriksa perubahan dari sisi:

* correctness;
* readability;
* maintainability;
* error handling;
* kemungkinan regression;
* test coverage;
* masalah security dasar.

Contoh:

```text
/code-review Review seluruh perubahan fitur reset password
```

---

## `/orchestrate`

Digunakan untuk pekerjaan multi-agent yang benar-benar dibagi menjadi beberapa worker atau session.

Versi ECC terbaru lebih berorientasi ke:

* beberapa terminal atau pane;
* `dmux`;
* git worktree;
* agent yang mengerjakan bagian berbeda secara paralel.

Gunakan hanya jika tugas dapat dipisahkan secara jelas, misalnya:

```text
Worker 1: database dan migration
Worker 2: API endpoint
Worker 3: frontend
Worker 4: test
```

Jangan biarkan beberapa worker mengedit file yang sama karena rawan conflict.

Untuk sub-agent biasa dalam satu session OpenCode, prompt delegasi langsung lebih sederhana daripada `/orchestrate`.

# Skill yang digunakan untuk fitur reset password

## 1. `search-first`

Mencari implementasi, dependency, dan pola yang sudah tersedia sebelum membuat solusi baru.

Digunakan untuk:

* mengecek library email;
* mencari pola token reset yang sudah ada;
* mengecek authentication framework;
* menghindari membuat utility duplikat.

---

## 2. `coding-standards`

Aturan dasar kualitas code lintas project.

Membantu menjaga:

* penamaan;
* readability;
* struktur fungsi;
* immutability;
* error handling;
* KISS, DRY, dan YAGNI.

Skill ini cocok dijadikan baseline pada hampir semua task coding.

---

## 3. `backend-patterns`

Pola backend untuk service, repository, controller, database, middleware, caching, dan error handling.

Untuk reset password, skill ini membantu merancang:

* reset-token service;
* database query;
* email job;
* expiration token;
* middleware dan validation;
* transaction dan error handling.

---

## 4. `api-design`

Aturan desain endpoint REST yang konsisten.

Untuk reset password, misalnya:

```text
POST /api/auth/forgot-password
POST /api/auth/reset-password
```

Skill ini membantu menentukan:

* request dan response;
* status code;
* error format;
* validation;
* versioning;
* rate limiting.

---

## 5. `security-review`

Skill penting untuk fitur authentication dan data sensitif.

Untuk reset password, skill ini harus memeriksa:

* token tidak disimpan sebagai plaintext;
* token memiliki expiration;
* token hanya dapat dipakai sekali;
* response tidak membocorkan apakah email terdaftar;
* endpoint memiliki rate limiting;
* session lama dapat di-revoke;
* password baru di-hash dengan benar;
* token tidak masuk log.

---

## 6. `tdd-workflow`

Mengatur implementasi menggunakan test terlebih dahulu.

Test yang relevan:

* email valid;
* email tidak terdaftar;
* token valid;
* token salah;
* token kedaluwarsa;
* token sudah digunakan;
* password tidak memenuhi policy;
* rate limit;
* reset berhasil;
* session lama tidak lagi berlaku.

---

## 7. `e2e-testing`

Menggunakan Playwright untuk menguji user flow sebenarnya.

Contoh flow:

```text
buka forgot password
→ masukkan email
→ ambil token dari mock email
→ buka halaman reset
→ masukkan password baru
→ login dengan password baru
```

Gunakan untuk fitur yang melibatkan browser, form, routing, dan beberapa endpoint.

---

## 8. `verification-loop`

Digunakan pada tahap akhir untuk memastikan:

* project berhasil di-build;
* typecheck lolos;
* lint lolos;
* unit test lolos;
* integration test lolos;
* E2E test lolos.

Agent tidak boleh menyatakan task selesai sebelum verification berhasil atau menjelaskan kegagalan yang belum dapat diperbaiki.

---

## 9. `frontend-patterns`

Aktifkan jika fitur memiliki halaman atau komponen frontend.

Membantu mengatur:

* React component;
* state;
* form validation;
* loading dan error state;
* data fetching;
* accessibility;
* responsive UI.

---

## 10. `strategic-compact`

Berguna untuk task panjang dengan banyak fase.

Gunakan setelah:

* exploration selesai;
* planning selesai;
* satu milestone selesai;
* context mulai terlalu besar.

Tujuannya menjaga informasi penting tanpa membawa seluruh hasil pencarian lama ke tahap implementasi.

---

## 11. `dmux-workflows`

Digunakan hanya untuk multi-agent eksternal yang benar-benar berjalan paralel.

Cocok ketika pekerjaan dapat dibagi ke folder atau concern berbeda:

```text
Agent A: backend
Agent B: frontend
Agent C: tests
Agent D: security review
```

Lebih aman menggunakan git worktree untuk setiap agent.

# Agent yang dapat digunakan

## Agent bawaan OpenCode

### `explore`

Agent read-only untuk mencari file dan memahami codebase.

Gunakan untuk:

```text
@explore Cari seluruh authentication flow, user model,
email service, dan pola validation yang sudah ada.
```

### `scout`

Agent read-only untuk memeriksa dokumentasi dan source dependency eksternal.

Gunakan untuk:

```text
@scout Periksa dokumentasi library authentication yang digunakan
project ini dan cari cara reset password yang direkomendasikan.
```

### `general`

Sub-agent serbaguna yang dapat mengerjakan task terpisah.

Gunakan untuk pekerjaan paralel yang tidak menyentuh file yang sama.

---

## Agent ECC

Agent berikut tersedia jika paket agent ECC ikut terpasang:

### `planner`

Membuat rencana implementasi.

### `architect`

Menentukan desain, batas antar-module, dan trade-off.

### `tdd-guide`

Menyusun test dan menjaga workflow TDD.

### `security-reviewer`

Melakukan audit security khusus.

### `code-reviewer`

Mereview kualitas dan correctness perubahan.

Jika agent ECC tertentu tidak tersedia, gunakan agent bawaan:

```text
planner/architect → plan atau general
researcher        → scout
code exploration  → explore
implementation    → general
```

# Prompt rekomendasi: satu session dengan banyak sub-agent

```text
Tambahkan fitur reset password end-to-end.

Gunakan skill yang relevan:
- search-first
- coding-standards
- backend-patterns
- api-design
- security-review
- tdd-workflow
- e2e-testing
- verification-loop

Workflow:

1. Gunakan explore untuk memetakan authentication flow, database,
   email service, routing, validation, dan pola test yang sudah ada.

2. Gunakan scout jika perlu memeriksa dokumentasi dependency yang
   digunakan project.

3. Buat desain implementasi. Jangan mengubah arsitektur project
   tanpa alasan kuat. Ikuti pattern yang sudah ada.

4. Delegasikan task independen ke beberapa sub-agent jika aman.
   Jangan izinkan dua agent mengedit file yang sama.

5. Implementasikan:
   - endpoint forgot password;
   - pembuatan token aman;
   - penyimpanan hash token dan expiration;
   - pengiriman email;
   - halaman reset password;
   - penggunaan token satu kali;
   - invalidasi token;
   - rate limiting.

6. Gunakan tdd-workflow. Buat unit dan integration test sebelum
   menyelesaikan implementasi.

7. Setelah implementasi, jalankan security-reviewer dan
   code-reviewer. Kedua review dapat berjalan paralel karena hanya
   membaca perubahan.

8. Perbaiki seluruh temuan critical dan high priority.

9. Jalankan verification-loop:
   - build;
   - typecheck;
   - lint;
   - unit test;
   - integration test;
   - E2E test.

10. Jangan berhenti hanya pada planning. Selesaikan implementasi
    sampai verification lolos.

11. Pada laporan akhir tampilkan:
    - file yang diubah;
    - keputusan desain;
    - test yang ditambahkan;
    - hasil build/lint/typecheck/test;
    - temuan security;
    - risiko atau pekerjaan yang masih tersisa.
```

# Prompt untuk multi-agent yang benar-benar paralel

```text
/orchestrate Implementasikan fitur reset password end-to-end.

Pisahkan pekerjaan menggunakan worker dan git worktree:

- Worker backend:
  database, token service, email service, dan API.

- Worker frontend:
  halaman forgot password dan reset password.

- Worker tests:
  unit, integration, dan E2E tests.

- Worker reviewer:
  security review dan code review setelah worker implementasi selesai.

Aturan:
- setiap worker menggunakan worktree berbeda;
- jangan mengedit file yang sama secara paralel;
- setiap worker menulis handoff dan daftar file yang diubah;
- integrasikan hasil satu per satu;
- selesaikan merge conflict di orchestrator;
- jalankan build, lint, typecheck, dan seluruh test setelah integrasi.
```
